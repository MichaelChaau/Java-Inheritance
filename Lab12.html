3. Read and Understand the Lab12 Code

Open the Lab12Code folder and start BlueJ. Read through the code and be sure you understand how it works.

  • Student.java represents a student, with a first name, a last name, a student id, a GPA, and an ArrayList of courses attempted. There is a constructor that creates a new student given first name, last name and id. (A new student does not have a GPA or courses yet.) There are set and get methods for first name, last name. id is a read-only field and there is no set method. (After all, once you get your ID number, it cannot be changed). There is no set method for GPA, since gpa is adjusted automatically whenever a student completes a course. There are also methods to add courses and list courses. Finally, there is a toString() method to print a Student object.

    The Student class also contains a private inner class called CourseAttempted that represents a course that has been taken by a student. An attempted course contains a catalog Course, with the semester and year it was taken, and the grade received. Student class uses this class as a helper class. No other classes need access to it, but it is useful to the Student class.

  • Staff.java represents a staff member, with a first name, a last name, an employee id, a telephone extension, a salary and number of years of service. There is a constructor that creates a new staff member given first name, last name, id, extension, salary and years of service. Go over the Staff.java file to understand its methods.

  • Faculty.java represents a faculty member, with a first name, a last name, an employee id, a telephone extension, a salary, and a rank. Go over the Faculty.java file and understand its methods.

  • Course.java represents a course, with a department, course number, title, and credits. There is a constructor that creates a new course from all of that information. There are set and get methods for all course information, and there is a toString() method to print a Course object.

  • The Lab12 class contains a main method that creates two student objects, 2 Faculty objects and a Staff member and prints all of them. The Lab12 class is there primarily to let you see what the output from the toString() methods looks like. You will not use the Lab12 class in this lab, but you can run it to look at the output.

  • StudentTest.java is a JUnit test class with two test methods. It tests the constructor, addCourse() and toString() methods of the Student class by creating two Students, four Courses, adding the Courses to the Students and calling the toString(). Make sure you understand why expected value in assertEquals() method looks like it does.

  • StaffTest.java is a JUnit test class with one test method. It tests the constructor and toString() methods of the Staff class by creating a Staff member and calling the toString(). Make sure you understand why expected value in assertEquals() method looks like it does.

  • FacultyTest.java is a JUnit test class with one test method. It tests the constructor and toString() methods of the Faculty class by creating a Faculty member and calling the toString(). Make sure you understand why expected value in assertEquals() method looks like it does.

5. Refactoring Code and Unit Tests

For this lab, you are going to refactor the code to use inheritance. Refactoring is a process of rewriting existing code to make it better - more maintainable, or more efficient, or more readable - without changing its behavior. Having a unit test suite to run leaves you free to make these refactoring changes, while still being sure that the (tested) behavior of the code remains the same.

Every time you finish making changes to your code, you should re-run your test suite to make sure the code still works.

6. Refactor Into An Inheritance Hierarchy

When you read the code for Student, Staff, and Faculty you probably noticed a lot of duplicated code between the three classes. (If you didn’t, go back and look for that…) Duplicated code often means that the classes are related to each other through inheritance.

Figure out how to use inheritance to extract the common features to create a super class (maybe you need two super classes - Person and Employee?).

Design your inheritance hierarchy on paper first. Refer to Inheritance_Ex2.zip - the inheritance example posted in Piazza and in Blackbaord. Understand how the Employee class has the features common to all the classes. Write down the methods and data fields for each of the classes - Student, Staff and Faculty. Figure out how to pull out the common data and methods into one or more super classes. Make sure not to duplicate the data fields!

7. Modify The Code to Implement Your Inheritance Heirarchy

Modify the existing code to fit your design. You will have to create additional classes and move around the existing code to different classes. In this lab, you are not writing much extra code - you are basically rearranging the existing code to different classes.

Be sure to rerun your test suite each time you complete a set of changes to make sure the code still works. Deciding when you’ve made enough changes to test the code will take some experience. You do not need to write any additional test classes.

8. Clean Up Your Code Before Submitting

Once you’ve completed your refactoring, and all of tests pass, you are done.

Go back through the source code of all of your non-test classes and be sure:

  1. The class has a comment at the top with an accurate description
  2. The class’ comment reflects the author.
  3. Each method has an accurate comment.
